17  Pipeline Pattern

The event-driven system pattern (Chapter 10) introduces asynchronous communication based on events, which is effective for decoupling components and responding to state changes. However, some systems require more structured, sequential processing of data where transformations occur in a defined order.

The pipeline pattern provides this structured approach to data processing.

The pipeline pattern structures a system as a sequence of processing stages through which data flows progressively. Each stage performs a transformation on the data before passing the result to the next stage in the sequence.

The fundamental idea behind this pattern is incremental transformation. Instead of processing complex operations within a single component, the system decomposes processing into a chain of specialized stages. Each stage performs a specific function and operates independently of the internal logic of other stages.

This structural organization improves modularity, scalability, and clarity of data processing workflows. It is widely used in domains such as data engineering, machine learning pipelines, build systems, and stream processing platforms.

17.0.1 Processing Stages

A pipeline is composed of three fundamental types of elements.

Stages represent the processing units within the pipeline. Each stage receives input data, performs a transformation, and produces output data. The transformation may involve filtering, aggregation, enrichment, validation, or computation.

Stages are typically designed to be stateless or to maintain minimal local state so that they can be executed repeatedly and independently.

Transforms describe the operations performed by each stage. These operations define how input data is converted into output data. Transformations may involve format conversion, data enrichment, feature extraction, or rule-based processing.

Sinks represent the endpoints of the pipeline. A sink is responsible for consuming the final output of the pipeline and storing or delivering it to another system. Common examples include databases, file systems, dashboards, or downstream services.

Together, these elements form a directed processing chain in which data moves sequentially through the pipeline.

17.0.2 Batch and Streaming Pipelines

Pipelines can operate under different execution models depending on how data enters and moves through the system.

In batch pipelines, data is processed in discrete collections known as batches. A batch job reads a set of input data, processes it through the pipeline stages, and writes the results once processing completes. Batch pipelines are commonly used for periodic data processing tasks such as analytics, reporting, or large-scale data transformations.

In streaming pipelines, data is processed continuously as it arrives. Each event or data item flows through the pipeline stages in real time or near real time. Streaming pipelines are commonly used in event processing, real-time analytics, and monitoring systems.

The structural differences between batch and streaming pipelines often involve execution scheduling, state management, and resource allocation. Batch pipelines typically rely on job schedulers and periodic execution, while streaming pipelines rely on event triggers and long-running processing services.

Despite these differences, both models share the same fundamental architectural structure: a chain of stages connected by directed data flows.

17.0.3 Data Contracts and Schema Evolution

Because pipeline stages exchange structured data, the definition of data formats becomes a critical architectural concern.

Each stage expects input data that conforms to a particular structure and produces output data with a defined schema. These expectations form data contracts between stages.

If the schema changes unexpectedly, downstream stages may fail or produce incorrect results. For this reason, pipelines often rely on schema management mechanisms that track data structure versions and enforce compatibility rules.

Schema evolution strategies may include versioned schemas, backward-compatible field additions, or validation layers that ensure incoming data conforms to expected formats.

Maintaining explicit data contracts allows pipeline systems to evolve gradually while preserving the integrity of the processing workflow.

17.0.4 Pattern Signature

Within the Pattern Catalog framework, the pipeline pattern can be identified through a set of structural characteristics.

The architecture contains a chain of processing steps that operate sequentially on a flow of data. Each step represents a transformation stage within the pipeline.

Directed data flow edges connect these stages, indicating the movement of data from one processing stage to the next. The pipeline typically begins with one or more data sources and terminates at one or more sinks.

Triggers or scheduling mechanisms initiate the execution of the pipeline. In batch systems, these triggers may originate from job schedulers or time-based events. In streaming systems, they may originate from incoming data events.

The presence of a step chain, explicit data flow relationships between stages, and triggering mechanisms for execution collectively form the structural signature of the pipeline pattern.

The pipeline pattern provides a powerful abstraction for systems in which data processing can be decomposed into sequential transformations.

By organizing processing logic into independent stages connected by data flows, pipelines improve system modularity and make complex processing workflows easier to understand, scale, and evolve.

17.1 From Foundational Patterns to High-Leverage Patterns

The five patterns presented in Part III—Layered System, Modular Monolith, Microservices, Event-Driven System, and Pipeline—represent foundational architectural structures. These patterns are widely applicable and appear in many systems across diverse domains. They establish the structural vocabulary of system architecture.

However, architectural complexity often requires more specialized structures that address specific operational concerns or enable advanced capabilities. These structures are introduced in Part IV: High-Leverage System Patterns.

Part IV presents patterns that can directly populate blueprint structures and serve as “blueprint generators”—complex architectural units that provide significant structural and operational capabilities:

  • Workflow Engine Pattern (Chapter 12) provides structured orchestration of long-running distributed processes

  • Job Queue + Worker Pool Pattern (Chapter 13) enables scalable asynchronous task execution

  • Plugin System Pattern (Chapter 14) supports extensibility through external modules

  • Policy Engine Pattern (Chapter 15) centralizes authorization and rule evaluation

  • Multi-Tenant Platform Pattern (Chapter 16) enables serving multiple tenants within shared infrastructure

These patterns are more specialized than the foundational patterns in Part III, but they provide high leverage—each pattern introduces a complete subsystem architecture that can substantially shape system design. They demonstrate how pattern signatures and catalog schemas are applied to more complex architectural challenges.
