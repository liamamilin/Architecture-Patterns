31  Appendix A — Pattern Catalog Schema v0.1 (YAML / JSON)

This appendix defines the canonical schema used to represent entries in Pattern Catalog v0.1. The schema formalizes how architectural patterns are stored as structured knowledge objects that can be indexed, queried, and processed by architecture tooling.

The schema is intentionally designed to be human-readable, machine-usable, and ontology-aligned. It captures both the conceptual description of a pattern and the structural anchors required for computational analysis.

Two equivalent representations are provided:

  • YAML (recommended for authoring and documentation)

  • JSON (recommended for programmatic processing)

The schema is designed to integrate directly with the System Ontology and Blueprint Architecture framework defined in earlier volumes.

32 YAML Representation

pattern:
  id: string
  name: string
  type: structural | reliability | security | governance | extensibility
  level: micro | module | system | platform

  description:
    problem: string
    context: string
    forces: [string]
    consequences:
      benefits: [string]
      tradeoffs: [string]

  structure:
    components:
      - name: string
        type: ontology_entity
        role: string

    interfaces:
      - name: string
        type: api | message | extension | event
        producer: component
        consumer: component

    data:
      - name: string
        owner: component
        consistency: strong | eventual | append-only

    workflows:
      - name: string
        type: orchestration | event_flow | pipeline
        steps: [component]

  signature:
    entities:
      required: [ontology_entity]
      optional: [ontology_entity]

    relations:
      - source: ontology_entity
        target: ontology_entity
        type: relation_type
        strength: strong | weak

    interfaces:
      required: [interface_type]

    constraints:
      reliability: [string]
      scalability: [string]
      security: [string]

  variants:
    optional_slots:
      - name: string
        type: OR | XOR
        options: [string]

  tags:
    domain: [devtools, fintech, ai, infra, data]
    system_type: [web, cli, pipeline, agent, platform]
    quality: [scalable, reliable, secure, extensible]
    complexity: toy | production | enterprise

  compatibility:
    requires: [pattern_id]
    compatible_with: [pattern_id]
    conflicts_with: [pattern_id]

  ontology_hooks:
    entities: [ontology_entity]
    relations: [ontology_relation]

  evidence_model:
    signals:
      - type: dependency | config | runtime | repo_structure
        description: string

  confidence_model:
    strong_indicators: [string]
    weak_indicators: [string]

  examples:
    systems:
      - name: string
        description: string
        reference: url_or_repo

  anti_patterns:
    - name: string
      description: string

33.1 Design Notes

Pattern Catalog Schema v0.1 follows several principles:

Ontology alignment
All structural elements reference entities and relations defined in the System Ontology.

Computability
Pattern signatures are designed to support automated pattern detection and blueprint extraction.

Human readability
YAML is recommended for pattern authoring and documentation.

Extensibility
Future catalog versions may introduce additional fields without breaking compatibility.
