38  Appendix H — Pattern-to-Blueprint Projection Cheatsheet

(Slots → Views → Spec View)

This appendix summarizes how patterns contribute to blueprint construction and how blueprint structures are projected into different architectural views.

The Blueprint Architecture framework separates three layers:

Patterns
Reusable structural knowledge.

Blueprints
Concrete system architecture representations.

Views / Projections
Different perspectives derived from the blueprint structure.

Patterns do not directly generate documentation or diagrams. Instead, they populate the Blueprint Kernel, which then serves as the canonical source for all architectural views.

This appendix provides a concise mapping reference showing how pattern outputs propagate through the architecture representation pipeline.

39 Blueprint Kernel Slots

All blueprint representations are structured around the Blueprint Kernel.

Goal
Capabilities
Components
Interfaces
Data
Workflows

These six slots capture the minimal architecture of a system.

Patterns contribute structural elements that populate one or more of these slots.

40 Pattern Contribution to Blueprint Slots

Patterns generate recommended structural elements that populate kernel slots.

Pattern Element Blueprint Slot
problem solved Goal
provided capability Capabilities
structural components Components
communication contracts Interfaces
state or information structures Data
coordination logic Workflows

Example: Workflow Engine Pattern

Pattern Element Blueprint Slot
task orchestration capability Capabilities
orchestrator / workers / queue Components
task execution API Interfaces
workflow state store Data
DAG execution logic Workflows

41 Slot-to-View Projection

Blueprints are not consumed directly by users. Instead, different stakeholders require different views of the architecture.

Views are projections derived from the kernel slots.

Blueprint Slot Product View Architecture View Operations View Security View
Goal system objectives — — —
Capabilities feature modules logical services operational features access scope
Components feature components component graph deployment units trust boundaries
Interfaces user APIs service interfaces network endpoints authorization points
Data product data models storage topology persistence infrastructure data classification
Workflows user flows control flows runtime processes policy enforcement

Each view extracts a different subset of blueprint structure.

42 Product View Projection

The Product View focuses on user-visible capabilities.

Projection rules:

Capabilities → product features
Components → functional modules
Workflows → user flows
Data → product data entities

Example output:

  • system features

  • capability map

  • feature-to-component mapping

This view is primarily used by product designers and stakeholders.

43 Architecture View Projection

The Architecture View focuses on structural relationships within the system.

Projection rules:

Components → architecture nodes
Interfaces → communication edges
Data → storage nodes
Workflows → control flow edges

Example outputs:

  • component diagrams

  • interface graphs

  • data flow diagrams

This is the primary view used by system architects.

44 Operations View Projection

The Operations View focuses on runtime operation and infrastructure.

Projection rules:

Components → deployment units
Interfaces → network endpoints
Data → storage infrastructure
Workflows → runtime processes

Example outputs:

  • deployment topology

  • runtime execution flows

  • operational dependencies

This view supports infrastructure engineers and operators.

45 Security View Projection

The Security View focuses on trust boundaries and policy enforcement.

Projection rules:

Components → trust zones
Interfaces → security gateways
Data → classified resources
Workflows → policy enforcement paths

Example outputs:

  • security boundary diagrams

  • authentication and authorization flows

  • sensitive data access paths

This view supports security architecture analysis.

46 Spec View Projection

Unlike other views, the Spec View does not represent system structure directly.

Instead, it derives requirements and acceptance criteria from blueprint elements.

Projection rules:

Blueprint Element Spec Artifact
Goal high-level requirements
Capabilities feature requirements
Components system modules
Interfaces API contracts
Data data schema definitions
Workflows behavioral scenarios

Example Spec View output:

Requirement:
The system shall support asynchronous task orchestration.

Acceptance Criteria:
- tasks are scheduled through a workflow engine
- task state is persisted
- failed tasks are retried automatically

The Spec View is therefore a projection of architecture, not a separate design artifact.

47 Pattern → Blueprint → Views Pipeline

The full architecture representation pipeline can be summarized as follows.

Pattern Library
        ↓
Pattern Selection
        ↓
Blueprint Construction
        ↓
Blueprint Kernel
        ↓
View Projections
(Product / Architecture / Ops / Security)
        ↓
Spec View (derived requirements)

Patterns supply reusable structural knowledge.
Blueprints organize that knowledge into system architectures.
Views present those architectures to different stakeholders.

48 Quick Reference Summary

Stage Representation
Pattern reusable structural solution
Blueprint concrete system architecture
Kernel minimal architecture slots
Views stakeholder perspectives
Spec View derived requirements

This projection pipeline ensures that architecture knowledge remains consistent across design artifacts.

All documentation, diagrams, and specifications ultimately originate from the same blueprint structure.

49 Role in the Blueprint Architecture Framework

The projection model completes the architecture knowledge loop introduced throughout the Blueprint Architecture series.

Patterns provide reusable architectural structures.

Blueprints assemble those structures into system architectures.

Projection rules transform blueprint structures into multiple architectural views and derived specifications.

Together, these elements form a coherent architecture knowledge system in which design, documentation, and analysis all originate from the same structured representation.
